## 1. Feature Context

### 1.1 Overview

Ensures that projects generated by `hai3 create` are immediately usable out of the box â€” passing their own architecture validation, including a working CI workflow, and having consistent version constraints across engine, package manager, and .nvmrc configurations.

Problem: CLI-generated projects fail `npm run arch:check` on first run, lack CI configuration, and have engine constraints that may not align with the .nvmrc or bundled Node version. Users must manually fix these issues before they can develop.

Primary value: A freshly scaffolded project passes all its own checks without any manual intervention.

### 1.2 Purpose

Close the gap between what the CLI generates and what a production-ready project requires by ensuring generated projects include CI automation, pass architecture validation, and have a self-consistent version configuration.

**Requirements**: `cpt-frontx-fr-cli-templates`, `cpt-frontx-fr-cli-e2e-verification`

## 3. Processes / Business Logic (CDSL)

### Generate CI Workflow

- [x] `p1` - **ID**: `cpt-frontx-algo-cli-scaffolding-quality-generate-ci`

Includes a CI workflow file in generated projects, assembled from a template stored in `template-sources/project/configs/_github/workflows/ci.yml`.

**Input**: Package manager selection, .nvmrc version

**Output**: `.github/workflows/ci.yml` in the generated project

1. [x] - `p1` - Store CI workflow template in `template-sources/project/configs/_github/workflows/ci.yml` with underscore prefix to prevent GitHub from treating it as a monorepo workflow - `inst-store-ci-template`
2. [x] - `p1` - During template build (`copy-templates.ts`), rename `_github/` to `.github/` alongside existing `_pre-commit-config.yaml` rename - `inst-rename-github-dir`
3. [x] - `p1` - Include `.github/workflows/ci.yml` in `ROOT_CONFIG_FILES` in the project generator so it is copied to generated projects - `inst-include-ci-in-root-configs`
4. [x] - `p1` - CI template uses `node-version-file: '.nvmrc'` instead of hardcoded Node version - `inst-ci-node-from-nvmrc`
5. [x] - `p1` - CI template triggers on `push` and `pull_request` to `main` branch - `inst-ci-main-branch`
6. [x] - `p1` - Transform `cache: 'npm'` to target package manager via `transformPackageManagerText` for pnpm/yarn projects - `inst-ci-cache-transform`

### Fix Architecture Check for Generated Projects

- [x] `p1` - **ID**: `cpt-frontx-algo-cli-scaffolding-quality-fix-arch-check`

Ensures `npm run arch:check` passes on freshly generated projects by removing checks for scripts that are not included in the generated project and adding graceful degradation for Node version requirements.

**Input**: Generated project with `scripts/test-architecture.ts`

**Output**: Architecture check passes without errors

1. [x] - `p1` - Remove `generate:colors` from standalone architecture checks in `test-architecture.ts` since the script is excluded from generated projects - `inst-remove-generate-colors-check`
2. [x] - `p1` - Gate `arch:deps` (dependency-cruiser) behind a Node version check â€” skip with a warning when Node major version is below the minimum required - `inst-gate-arch-deps-node-version`

## 5. Definitions of Done

### CI Workflow Generation

- [x] `p1` - **ID**: `cpt-frontx-dod-cli-scaffolding-quality-ci-workflow`

The CLI **MUST** generate a `.github/workflows/ci.yml` file in every new project that runs architecture validation and build checks on push and pull request.

**Implements**:

- `cpt-frontx-algo-cli-scaffolding-quality-generate-ci`

### Architecture Check Passes on Fresh Projects

- [x] `p1` - **ID**: `cpt-frontx-dod-cli-scaffolding-quality-arch-check-pass`

A freshly generated project **MUST** pass `npm run arch:check` without any manual modifications after running the package manager install.

**Implements**:

- `cpt-frontx-algo-cli-scaffolding-quality-fix-arch-check`

### Package Manager Consistency in CI

- [x] `p1` - **ID**: `cpt-frontx-dod-cli-scaffolding-quality-pm-ci-consistency`

The generated CI workflow **MUST** use the same package manager selected during project creation â€” including the `cache` directive in `setup-node` and install/build commands.

**Implements**:

- `cpt-frontx-algo-cli-scaffolding-quality-generate-ci`

## 6. Acceptance Criteria

- [x] `frontx create my-app` with default options produces a project where `npm run arch:check` passes
- [x] Generated project includes `.github/workflows/ci.yml`
- [x] CI workflow uses Node version from `.nvmrc`, not a hardcoded value
- [x] `frontx create my-app --package-manager pnpm` produces a CI workflow with `cache: 'pnpm'` and pnpm commands
- [x] `frontx create my-app --package-manager yarn` produces a CI workflow with `cache: 'yarn'` and yarn commands
- [x] E2E PR smoke test validates presence of `.github/workflows/ci.yml` in generated projects
